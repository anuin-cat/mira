# Mira 前端视觉 Token

本文档定义 Mira 编辑器界面的基础视觉 token 和组合规则。它不是现有 CSS 数值清单，也不是组件说明书；遇到新 UI 时，优先用这些 token 组合出形态，只有明确必要时才保留组件级微调。

## 什么时候看

- 新增或调整按钮、输入框、菜单、弹层、工具栏、侧栏、分隔线时。
- 修改 `src/styles/themes.css`、`src/App.css`、`src/styles/mdx-editor.css` 中的基础视觉样式时。
- 不确定一个新尺寸、圆角、阴影或颜色是否应该引入时。

## 基本气质

Mira 是 mac 优先的本地写作工具，界面应接近 macOS 原生生产力应用：安静、紧凑、轻质、低干扰。优先用背景层级、细边界、圆角和克制留白建立结构，避免厚重卡片、大阴影、高饱和装饰和营销式大字号。

## Token

### 间距

基础间距只保留这组：

| Token | 值 | 用途 |
| --- | ---: | --- |
| `space-1` | `2px` | 图标组极窄间隔。 |
| `space-2` | `4px` | 分隔线外边距、微调间隔。 |
| `space-3` | `6px` | 紧凑控件内部间距、工具组间距。 |
| `space-4` | `8px` | 菜单/弹层内边距、按钮水平内边距下限。 |
| `space-5` | `12px` | 工具栏和侧栏头部水平留白。 |
| `space-6` | `16px` | 普通面板内容留白下限。 |
| `space-7` | `20px` | 侧栏内容标准留白。 |
| `space-8` | `24px` | 内容区大留白、空状态。 |

`44px`、`60px`、`120px` 这类阅读区留白是编辑器正文布局常量，不作为通用间距 token。

### 控件尺寸

| Token | 值 | 用途 |
| --- | ---: | --- |
| `control-xs` | `24px` | 极紧凑输入框、小型内嵌控件。 |
| `control-sm` | `28px` | 图标按钮、菜单项、紧凑列表操作。 |
| `control-md` | `32px` | 标准输入框、选择器、文件树行高。 |
| `bar-height` | `44px` | 顶部工具栏、侧栏头部。 |

图标尺寸只保留三档：

| Token | 值 | 用途 |
| --- | ---: | --- |
| `icon-sm` | `14px` | 辅助动作、小型弹层按钮。 |
| `icon-md` | `16px` | 默认图标。 |
| `icon-lg` | `18px` | 工具栏或更强调的操作。 |

现有 `15px`、`17px`、`19px` 等只属于组件级视觉微调，不应升级成全局 token。

### 布局常量

| 对象 | 建议值 |
| --- | --- |
| 文件侧栏 | `160px` 最小，`280px` 默认，`480px` 最大 |
| AI 侧栏 | `290px` 最小，`380px` 默认，`560px` 最大 |
| 编辑器正文最大宽度 | `960px` |
| macOS 风格菜单宽度 | 约 `170px` |
| 普通浮层最大宽度 | `min(420px, calc(100vw - 32px))` |

这些是布局语义值，不要混入普通控件尺寸。

### 圆角

| Token | 值 | 用途 |
| --- | ---: | --- |
| `radius-xs` | `4px` | 极小输入框、细节控件。 |
| `radius-sm` | `6px` | 侧栏按钮、列表行、文件树节点。 |
| `radius-md` | `8px` | 标准按钮、输入框、普通容器。 |
| `radius-lg` | `12px` | 普通弹层、工具栏溢出面板。 |
| `radius-xl` | `16px` | macOS 小浮层、上下文菜单近似值。 |
| `radius-panel` | `22px` | 编辑器主面板外轮廓、消息气泡。 |
| `radius-pill` | `999px` | pill、线性指示器、拖放落点线。 |

原则：普通控件不做大胶囊；菜单当前实现可接近 `17px`，但文档层面按 `radius-xl` 管理。

### 字体

全局继续使用系统字体：

```css
font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', Helvetica, sans-serif;
```

| Token | 值 | 用途 |
| --- | ---: | --- |
| `text-xs` | `12px` | 快捷键、辅助说明、元信息。 |
| `text-sm` | `13px` | 菜单项、紧凑控件。 |
| `text-md` | `14px` | 默认 UI 文本。 |
| `text-editor` | `14/15/16/18px` | 编辑器正文随字体档位变化。 |

紧凑界面默认用 `13-14px`。除编辑器正文标题外，不引入大标题体系。

### 颜色

颜色优先来自 `src/styles/themes.css`：

- 背景：`--theme-bg-primary`、`--theme-bg-sidebar`、`--theme-bg-toolbar`
- 文本：`--theme-text-primary`、`--theme-text-secondary`、`--theme-text-tertiary`
- 边界：`--theme-border`、`--theme-border-sidebar`
- 状态：`--theme-hover-bg`、`--theme-active-bg`、`--theme-selection`、`--theme-accent`
- 控件：`--theme-editor-control-*`
- 编辑正文：`--theme-editor-link-*`、`--theme-editor-inline-code-*`、`--theme-editor-blockquote-*`

允许固定色的场景要少：macOS 菜单高亮蓝、亮色磨砂菜单的半透明文字、临时危险态红色。新增固定色前应先判断是否该变成 theme token。

## 组合规则

### 按钮

- 图标按钮：`control-sm` 或 `control-md` 正方形，图标用 `icon-md` 或 `icon-lg`。
- 普通按钮：高度优先 `control-md`，圆角 `radius-md`。
- 工具栏按钮：`30px` 可作为现有工具栏微调保留；新增工具栏按钮优先靠近 `control-md`。
- hover 使用主题弱背景，focus-visible 使用 `--theme-selection` 派生的 2px ring。

### 菜单与列表行

- 菜单项和紧凑列表行高度优先 `control-sm`。
- 菜单项字号 `text-sm`，图标 `icon-md`。
- 带图标菜单使用三列：图标、文本、快捷键；文本列必须可省略。
- 分隔线为 `1px`，带图标菜单的分隔线从文本列开始缩进。

### 输入框与选择器

- 默认高度 `control-md`，小型弹层内可用 `control-xs` 或 `control-sm`。
- 圆角优先 `radius-md`，极小输入框可用 `radius-xs`。
- 不要同时用大高度和大垂直 padding 堆出控件。

### 弹层

- 普通弹层：`radius-lg`、`space-4` 内边距、`1px` 控件边框、轻阴影。
- macOS 风格上下文菜单：宽约 `170px`、`radius-xl`、半透明磨砂、较强浮层阴影。
- 弹层与触发器保持 `6-8px` 间距。
- 弹层层级必须明确；MDXEditor portal 外壳当前需要高于 sticky 工具栏。

### 分栏与边界

- 面板之间优先靠背景差异、细边界和圆角分层，不用厚边框。
- 主编辑面板外圆角使用 `radius-panel`。
- 拖拽命中区可大于可见线：命中宽度 `8px`，可见线 `1px`，默认透明，hover/focus 时显示。
- 避免相邻面板双边框。

## 边框、阴影、动效

- 边框默认 `1px`。弱边界用 `color-mix(in srgb, var(--theme-border) 50-60%, transparent)`。
- 面板本体尽量无阴影或极弱阴影；明确阴影主要给浮层。
- 阴影分三档即可：按钮 hover 轻阴影、普通弹层阴影、上下文菜单/磨砂浮层阴影。
- 颜色和背景过渡使用 `0.1-0.18s ease`；面板进入可用 `0.25-0.3s cubic-bezier(0.4, 0, 0.2, 1)`。
- hover 不做弹跳和明显位移。

## 例外处理

以下数值可以保留为组件级实现细节，但不要推广为全局 token：

- 右键菜单的 `172px` 宽度、`17px` 圆角、菜单内局部 `15/17/19px` 微调。
- MDXEditor 或 shadcn 组合为了对齐第三方内部结构产生的特殊宽度、偏移和层级。
- 编辑器正文的 `44px 60px 120px` 阅读留白。

如果新增了例外，必须能说明它服务于具体组件的视觉对齐、第三方限制或阅读体验；否则应回到基础 token。

## 何时修改规则

修改或新增本规则前，先判断问题属于“复用现有 token”、“组件级例外”还是“应升级为规则”：

- **优先复用**：只是某个组件看起来不合适，但能通过现有 token 的组合解决时，不修改本文档。
- **组件级例外**：只服务一个具体组件、第三方库对齐或阅读体验的数值，留在组件实现中，并在“例外处理”补一句说明即可。
- **升级为规则**：同一视觉需求在两个以上独立场景重复出现，或现有 token 无法表达一种稳定的新语义时，才新增 token 或组合规则。
- **替换旧规则**：如果新规则会让旧规则失去意义，应替换旧规则，不要并列保留两个相近 token。
- **删减规则**：当某个 token 或规则不再被使用，或只是历史实现细节，应从本文档删除。

新增规则必须满足三点：能说明用途边界、能被多个场景复用、不会和现有 token 形成近似重复。不要因为一次局部视觉微调就新增全局 token。

## 检查清单

- 新数值是否能归入现有 token；如果不能，是否只是组件级例外。
- 新颜色是否能使用 theme token。
- 控件高度是否落在 `24/28/32/44px` 体系内。
- 圆角是否落在 `4/6/8/12/16/22/999px` 体系内。
- 边界是否足够轻，是否避免了双线和厚线。
- hover、active、focus-visible、disabled 是否完整。
- 长文本、长路径是否能省略。
- 默认主题和至少一个深色主题下是否仍可读。
